# groupSectors_simulatedAnnealing_1 算法时间复杂度分析

## 一、算法整体结构

该算法使用模拟退火（Simulated Annealing, SA）解决扇区分组问题，目标是将冲突的扇形分配到不同组中。

## 二、各阶段时间复杂度详细分析

### 1. 构建冲突图 (1282-1289 行)

```cpp
for (int i = 0; i < n; ++i) {
    for (int j = i + 1; j < n; ++j) {
        if (sectorsIntersect(sectors[i], sectors[j])) {
            adj[i][j] = adj[j][i] = true;
        }
    }
}
```

**时间复杂度**: O(n² × C_intersect)

- 外层循环: n 次
- 内层循环: (n-1) + (n-2) + ... + 1 = n(n-1)/2 ≈ n²/2
- `sectorsIntersect` 函数: 几何计算，涉及点包含检查、线段相交检测等，但操作数量固定，可视为 O(1)
- **总计**: O(n²)

### 2. sectorsIntersect 函数复杂度 (936-1008 行)

该函数执行以下操作：

- 特殊情况检查: O(1)
- 点包含检查: 最多 6 次 pointInSector 调用，每次 O(1)
- 线段相交检测: 最多 4 次 segmentsIntersect 调用，每次 O(1)

**时间复杂度**: O(1) (常数时间)

### 3. 初始解生成 (1360-1405 行)

#### 3.1 排序阶段 (1365-1378 行)

```cpp
std::sort(sector_order.begin(), sector_order.end(), ...)
```

**时间复杂度**: O(n log n)

#### 3.2 贪心分配阶段 (1381-1405 行)

```cpp
for (int idx : sector_order) {  // n 次循环
    for (int g = 0; g < num_groups_initial; ++g) {  // 最多 n 组
        for (int j = 0; j < n; ++j) {  // n 次检查
            if (current_solution[j] == g && adj[idx][j]) {
                conflicts++;
            }
        }
    }
}
```

**时间复杂度**: O(n³)

- 外层循环: n 次
- 中层循环: 最坏情况 n 组
- 内层循环: n 次冲突检查

### 4. 模拟退火主循环 (1418-1508 行)

#### 4.1 外层循环次数

```cpp
const double initial_temp = 50.0;
const double final_temp = 0.1;
const double cooling_rate = 0.98;
```

温度从 50.0 降到 0.1，每次乘以 0.98：

- 循环次数 = log(50/0.1) / log(1/0.98) ≈ log(500) / log(1.0204) ≈ 6.215 / 0.0202 ≈ **308 次**

#### 4.2 内层循环次数

```cpp
const int iters_per_temp = std::max(100, 5 * n);
```

- 当 n < 20 时: 100 次
- 当 n ≥ 20 时: 5n 次

#### 4.3 每次迭代的复杂度

**a) 邻域生成策略 (1425-1481 行)**

**策略 1: 随机移动 (70%概率)**

```cpp
// 计算可移动的组
for (int g = 0; g < num_groups_initial; ++g) {  // 最多 n 组
    for (int j = 0; j < n; ++j) {  // n 次检查
        if (j != sector_to_move && new_solution[j] == g && adj[sector_to_move][j]) {
            can_move = false;
            break;
        }
    }
}
```

**时间复杂度**: O(n²)

**策略 2: 交换两个扇形 (30%概率)**

```cpp
// 检查交换后是否无冲突
for (int j = 0; j < n; ++j) {  // n 次检查
    if (j != sector1 && adj[sector1][j] && new_solution[j] == new_solution[sector2]) {
        valid_swap = false;
        break;
    }
    if (j != sector2 && adj[sector2][j] && new_solution[j] == new_solution[sector1]) {
        valid_swap = false;
        break;
    }
}
```

**时间复杂度**: O(n)

**平均邻域生成复杂度**: 0.7 × O(n²) + 0.3 × O(n) = **O(n²)**

**b) 能量函数计算 (1306-1358 行)**

```cpp
// 计算组数: O(n)
std::set<int> unique_groups(solution.begin(), solution.end());

// 计算冲突数: O(n²)
for (int i = 0; i < n; ++i) {
    for (int j = i + 1; j < n; ++j) {
        if (adj[i][j] && solution[i] == solution[j]) {
            num_conflicts++;
        }
    }
}

// 计算方差: O(n)
// AUV优先级: O(n)
```

**时间复杂度**: O(n²)

**c) Metropolis 准则**: O(1)

**单次迭代总复杂度**: O(n²) + O(n²) + O(1) = **O(n²)**

#### 4.4 模拟退火总复杂度

- 外层循环: 308 次
- 内层循环: max(100, 5n) 次
- 单次迭代: O(n²)

**总复杂度**: O(308 × max(100, 5n) × n²)

**当 n ≥ 20 时**: O(308 × 5n × n²) = **O(1540 × n³) ≈ O(n³)**

**当 n < 20 时**: O(308 × 100 × n²) = **O(30800 × n²) ≈ O(n²)**

### 5. 结果转换 (1512-1535 行)

```cpp
for (int i = 0; i < n; ++i) {
    if (best_solution[i] >= 0 && best_solution[i] < final_num_groups) {
        groups[best_solution[i]].push_back(sectors[i]);
    }
}
```

**时间复杂度**: O(n)

### 6. 优化组函数 (1596-1624 行)

```cpp
while (improved) {
    for (int i = 0; i < groups.size(); ++i) {  // 最多 n 组
        for (int j = i + 1; j < groups.size(); ++j) {  // 最多 n 组
            for (const auto& s1 : groups[i]) {  // 平均 n/组数
                for (const auto& s2 : groups[j]) {  // 平均 n/组数
                    if (sectorsIntersect(s1, s2)) {  // O(1)
                        can_merge = false;
                        break;
                    }
                }
            }
        }
    }
}
```

**最坏情况时间复杂度**: O(n⁴)

- 但实际中组数通常远小于 n，且合并操作会减少组数
- **实际复杂度**: O(n²) 到 O(n⁴) 之间，取决于冲突图的密度

## 三、整体时间复杂度总结

### 最坏情况 (n ≥ 20)

| 阶段           | 复杂度                     |
| -------------- | -------------------------- |
| 构建冲突图     | O(n²)                      |
| 初始解生成     | O(n log n) + O(n³) = O(n³) |
| 模拟退火主循环 | O(n³)                      |
| 结果转换       | O(n)                       |
| 优化组         | O(n²) ~ O(n⁴)              |

**总时间复杂度**: **O(n⁴)** (由优化组函数主导)

### 实际运行情况 (n ≥ 20)

如果优化组函数执行时间较短，主要复杂度来自：

- **O(n³)** (模拟退火主循环)

### 空间复杂度

- 冲突图邻接矩阵: O(n²)
- 解向量: O(n)
- 组向量: O(n)
- **总空间复杂度**: O(n²)

### 4.2 计算时间估算

假设 n = 30 (30 个扇形请求):

#### 4.2.1 构建冲突图

- 操作次数: 30 × 29 / 2 = 435 次 sectorsIntersect 调用
- 每次调用: ~100 条指令 (几何计算)
- 总指令数: 43,500 条
- **STM32F4 (168 MHz)**: 43,500 / 168,000,000 ≈ **0.26 ms**
- **STM32L4 (80 MHz)**: 43,500 / 80,000,000 ≈ **0.54 ms**

#### 4.2.2 初始解生成

- 排序: O(n log n) = 30 × log(30) ≈ 150 次比较
- 贪心分配: O(n³) = 30³ = 27,000 次操作
- 总操作: ~27,150 次
- **STM32F4**: 27,150 / 168,000,000 ≈ **0.16 ms**
- **STM32L4**: 27,150 / 80,000,000 ≈ **0.34 ms**

#### 4.2.3 模拟退火主循环

- 外层循环: 308 次
- 内层循环: max(100, 5×30) = 150 次
- 总迭代次数: 308 × 150 = 46,200 次
- 每次迭代: O(n²) = 900 次操作
- 总操作数: 46,200 × 900 = **41,580,000 次**

**STM32F4 (168 MHz)**:

- 计算时间: 41,580,000 / 168,000,000 ≈ **248 ms** (0.25 秒)

**STM32L4 (80 MHz)**:

- 计算时间: 41,580,000 / 80,000,000 ≈ **520 ms** (0.52 秒)

**MSP430 (16 MHz)**:

- 计算时间: 41,580,000 / 16,000,000 ≈ **2.6 秒**

#### 4.2.4 优化组函数

- 最坏情况: O(n⁴) = 30⁴ = 810,000 次操作
- **STM32F4**: 810,000 / 168,000,000 ≈ **4.8 ms**
- **STM32L4**: 810,000 / 80,000,000 ≈ **10 ms**

## 五、MCU 部署可行性结论

### ✅ 可行的场景

1. **节点数量 ≤ 30，使用 STM32F4 或更高性能 MCU**

   - 计算时间 < 300 ms
   - 满足实时性要求

2. **节点数量 ≤ 20，使用 STM32L4**
   - 计算时间 < 100 ms
   - 性能充足

### ⚠️ 需要优化的场景

1. **节点数量 30-50，使用 STM32L4 或更低性能 MCU**

   - 计算时间 0.5-1.5 秒
   - 可能影响实时性

2. **节点数量 > 50**
   - 计算时间 > 1 秒
   - 需要算法优化或硬件升级

### ❌ 不可行的场景

1. **节点数量 > 100**

   - 计算时间 > 9 秒
   - 严重超时，不适合实时系统

2. **使用超低功耗 MCU (如 MSP430) 处理大量节点**
   - 计算能力不足

## 六、优化建议

### 6.1 算法层面优化

1. **减少迭代次数**

   - 降低 `iters_per_temp`: 从 `max(100, 5*n)` 改为 `max(50, 3*n)`
   - 提高 `cooling_rate`: 从 0.98 改为 0.95 (加快收敛)
   - 降低 `max_no_improvement`: 从 1000 改为 500

2. **优化能量函数计算**

   - 增量更新能量值，而不是每次都完整计算
   - 复杂度从 O(n²) 降低到 O(n)

3. **简化邻域生成**

   - 优先使用交换策略 (O(n))，减少随机移动 (O(n²))

4. **提前终止**
   - 如果能量值连续多次无改进，提前退出

### 6.2 硬件层面优化

1. **使用浮点运算单元 (FPU)**

   - STM32F4 有硬件 FPU，可加速几何计算

2. **使用 DSP 指令集**

   - 加速向量运算

3. **增加 MCU 性能**
   - 升级到更高主频的 MCU

### 6.3 架构层面优化

1. **分布式计算**

   - 将计算任务分配到多个节点
   - 使用主节点 (Server) 进行计算，普通节点只负责数据收集

2. **预计算和缓存**

   - 预计算冲突图
   - 缓存常用计算结果

3. **简化算法**
   - 对于小规模问题 (n < 20)，使用简单的贪心算法代替模拟退火
   - 只在 n ≥ 20 时使用模拟退火

## 七、总结

### 时间复杂度

- **最坏情况**: O(n⁴)
- **实际运行 (n ≥ 20)**: O(n³)
- **空间复杂度**: O(n²)

### MCU 部署建议

1. **n ≤ 20**: 所有主流 MCU 都可行
2. **20 < n ≤ 30**: 需要 STM32F4 或更高性能 MCU
3. **30 < n ≤ 50**: 需要优化算法或使用高性能 MCU
4. **n > 50**: 建议使用分布式计算或简化算法

### 关键限制因素

1. **模拟退火主循环**是主要计算瓶颈
2. **能量函数计算** (O(n²)) 在每次迭代中执行
3. **迭代次数** (308 × max(100, 5n)) 随 n 线性增长

### 最终建议

对于**水声网络普通节点**，如果节点数量 ≤ 30，使用 **STM32F4 级别或更高性能的 MCU** 可以支撑计算。如果节点数量更多，建议：

1. 优化算法参数
2. 使用增量能量计算
3. 考虑分布式计算架构
4. 或使用简化的贪心算法
